Q1 Three-Schema (Three-Level) Architecture of DBMS

⚡ Core Intuition

Three layers hide complexity: users see only what they need, designers see the logical map, storage details stay buried.

📖 Concept Explained
  • External level — what each user sees (views).
  • Conceptual level — global logical structure of whole DB.
  • Internal level — physical storage (files, indexes, blocks).
  • Two mappings glue layers: external↔conceptual, conceptual↔internal.
  • Proposed by ANSI/SPARC, 1975.
🔗 Connects to Data Independence Schema Views DBA
📐 Exam Answer Structure
  • 1-line definition + name (ANSI/SPARC).
  • Draw 3-layer diagram with mappings.
  • Describe each level (4 bullets each).
  • Mention purpose: abstraction + data independence.
  • List 2 advantages.
💡 Example

University DB: student sees marks view; admin sees full grade table; on disk, rows are stored as B-tree blocks.

Definition

A 3-layer abstraction model (ANSI/SPARC, 1975) separating user views, conceptual schema, and physical storage.

Mnemonic — "VC IP": View → Conceptual → Internal → Physical DB.

✅ Model Exam Answer
  1. Define: ANSI/SPARC three-schema architecture (1975) for data abstraction.
  2. Draw the layered diagram (above).
  3. External level — multiple user views; hides irrelevant data.
  4. Conceptual level — global logical schema; one per DB.
  5. Internal level — physical storage, indexes, file layout.
  6. Two mappings translate between layers.
  7. Purpose: enables data independence, security, multi-user views.

📚 Primary: DBMS Module 1 (pp. 2–4) · Supplemented: Korth ch. 1.

Q2 Data Independence — Logical vs Physical

⚡ Core Intuition

Change storage without breaking apps. Add an index → SQL queries still work. Rename a column → views break.

📖 Concept Explained
  • Capacity to change schema at one level without changing higher level.
  • Physical DI — change internal schema, conceptual unaffected.
  • Logical DI — change conceptual schema, external views unaffected.
  • Physical DI is easy; logical DI is harder (apps depend on conceptual).
🔗 Connects to 3-Schema Architecture Mappings Schema Evolution

  • Comparison table (5 rows).

💡 Example

Physical DI — switch HDD to SSD or add index on roll_no; queries unchanged. Logical DI — add email column to Student; old views still return same data.

Pitfall

Don't swap the definitions. Physical = lower level. Logical = middle level.

Q3 DBMS Components & n-Tier Architecture

⚡ Core Intuition

Split the system into layers so each layer can scale, secure, and change independently. Centralized → 2-tier → 3-tier → n-tier.

📖 Concept Explained
  • 5 components: Hardware, Software, Data, Users, Procedures.
  • Centralized — everything on one mainframe.
  • 2-tier — client app talks directly to DB server.
  • 3-tier — client ↔ application server ↔ DB server.
  • n-tier — multiple middle layers (web, app, business, DB).

Different people touch the DB at different depths. DBA is the boss; end users just see forms.

Q5 Advantages of DBMS over File System

⚡ Core Intuition

File system = each app owns its own files → duplication, no rules, no sharing. DBMS = one shared brain with rules.

📖 Concept Explained
  • Redundancy control — one fact stored once.
  • Consistency — updates propagate via constraints.
  • Concurrency — many users without conflict (locks).
  • Security — role-based access, GRANT/REVOKE.
  • Backup & recovery — built-in via logs.
  • Data integrity — constraints (PK, FK, CHECK).
  • Sharing — multi-user, networked.

Q6 Data Models, Classification & DBMS Languages

⚡ Core Intuition

Models = blueprints. Relational won because tables + SQL are simple. Languages = verbs: define (DDL), change (DML), control (DCL), commit (TCL).

📖 Concept Explained
  • Data model categories — High-level (ER), Representational (Relational, Network, Hierarchical), Low-level (Physical).
  • Classification — single/multi-user, centralized/distributed, relational/NoSQL/OO/hierarchical.
  • DDL — CREATE, ALTER, DROP, TRUNCATE, RENAME.
  • DML — INSERT, UPDATE, DELETE, SELECT.
  • DCL — GRANT, REVOKE.
  • TCL — COMMIT, ROLLBACK, SAVEPOINT.
🔗 Connects to SQL Schema ER Model
📐 Exam Answer Structure
  • Define data model + list 4 types.
  • Draw taxonomy tree.
  • Mention DDL/DML/DCL/TCL with 2 commands each.
💡 Example

Banking DB: ER for design → Relational for tables → SQL DDL creates schema → DML inserts transactions → TCL commits transfer.

-- DDL
CREATE TABLE Student (id INT PRIMARY KEY, name VARCHAR(50));

-- DML
INSERT INTO Student VALUES (101, 'Ankit');
SELECT * FROM Student WHERE id = 101;

-- DCL
GRANT SELECT ON Student TO clerk_role;

-- TCL
BEGIN;
UPDATE Student SET name='Anki' WHERE id=101;
COMMIT;

Mnemonic — "DDDT": Define, Do, Delegate, Transact.

✅ Model Exam Answer
  1. Define data model + list 5 types (ER, relational, hierarchical, network, OO).
  2. Draw taxonomy tree.
  3. Brief 1-line per model.
  4. List 4 language categories with 2 commands each.
  5. SQL example snippet.

Q2 Explain the types of attributes in an ER model

⚡ Core Intuition

Attributes describe entities — but not all are equal. Some split, some compute, some repeat, some identify.

📖 Concept Explained

  • Composite — splits into parts (Name → First, Last).
  • Derived — computed from another (Age from DOB).
  • Multivalued — multiple values (Phone[1..n]).
  • Key — uniquely identifies an entity (RollNo).

Mnemonic — "SCD-MK": Simple, Composite, Derived, Multivalued, Key.

Q3 Explain types of keys in DBMS with examples

⚡ Core Intuition

PK = unique row stamp. FK = pointer to another table's stamp. Everything else is a subset story.

📖 Concept Explained
  • Super key — ANY set of attrs that's unique (may be bloated).
  • Candidate key — minimal super key (nothing wasted).
  • Primary key — one candidate picked by DBA.
  • Alternate key — remaining candidate keys.
  • Composite key — key built from 2+ columns.
  • Foreign key — references PK of another relation.

Q4 Explain cardinality & participation constraints

⚡ Core Intuition

Cardinality = "how many?" Participation = "must they?" One is a count, the other is a yes/no.

📖 Concept Explained
  • 1:1 — one to one (Person ↔ Passport).
  • 1:N — one to many (Dept → Employees).
  • M:N — many to many (Student ↔ Course).
  • Total participation — every entity must participate (double line).
  • Partial participation — some may not (single line).
🔗 Connects to ER diagramFK directionjunction table

💡 Example

1:N   → FK on the "many" side (Employee.DeptID)

Q5 Integrity constraints over relations

⚡ Core Intuition

Constraints are rules the data must always obey. They keep garbage out and pointers honest.

📖 Concept Explained
  • Domain — every value of an attribute lies in its declared type/range.
  • Key (uniqueness) — no two tuples agree on the key.
  • Entity integrity — primary key is NEVER NULL.
  • Referential integrity — FK either matches a PK value or is NULL.

Q6 Relational Model terminology — tuple, attribute, degree, cardinality, schema, instance

⚡ Core Intuition

A relation is just a table. The model just renames every part: rows = tuples, columns = attributes, blueprint = schema, snapshot = instance.

📖 Concept Explained
  • Tuple — single row.
  • Attribute — single column.

Q2 Cartesian Product & JOIN types (Theta, Equi, Natural, Outer)

⚡ Core Intuition

Cartesian × glues every row to every row; JOIN = × then σ; Natural drops the duplicate key column.

📖 Concept Explained
  • × — R(m rows) × S(n rows) = m·n rows.
  • θ-join — × + condition (any operator).
  • Equi-join — θ-join restricted to =.
  • Natural ⋈ — equi on same-name attrs, dedup column.
  • Outer — preserves unmatched rows with NULL.

💡 Example

Q1 Functional Dependency & Armstrong's Axioms

⚡ Core Intuition

If you know A, you can always look up B — that's A → B.

Armstrong's axioms are 3 inference rules (RAT) that let you derive new FDs from given ones.

📖 Concept Explained
  • FD X → Y: X uniquely determines Y in every tuple.
  • X: determinant (left side). Y: dependent (right side).
  • FDs encode real-world business rules — e.g. RollNo → Name.
  • Foundation of normalization — every NF is defined via FDs.
Definition

For relation R, X → Y holds iff for every pair of tuples t₁,t₂: if t₁[X]=t₂[X] then t₁[Y]=t₂[Y].

🔗 Connects to Closure F⁺ Candidate keys Normalization BCNF
📐 Exam Answer Structure
  • Definition + notation (X → Y)
  • Types: trivial, non-trivial, multivalued, transitive
  • Armstrong's axioms (RAT) — state + 1-line example each
  • Derived rules (union, decomposition, pseudo-transitivity)
  • Significance — basis for normalization
💡 Example

R(A,B,C). Given F = {A→B, B→C}. By transitivity: A→C ∈ F⁺.

🎯 Mnemonic — RAT: Reflexive, Augment, Transitive.
✅ Model Exam Answer
  1. Def: FD X→Y means X uniquely determines Y in R.
  2. Notation: X = determinant, Y = dependent. Diagram with arrow.
  3. Types: trivial (Y⊆X), non-trivial, multivalued, transitive.
  4. Armstrong (RAT): Reflexivity, Augmentation, Transitivity — state each.
  5. Derived: union, decomposition, pseudo-transitivity.
  6. Use: compute closures, find candidate keys, drive 2NF/3NF/BCNF.

Q2 Normalize a Table up to BCNF — Worked Example

⚡ Core Intuition

Normalization = remove redundancy by splitting tables. Each NF fixes one kind of bad dependency.

1NF fixes multi-valued cells. 2NF fixes partial deps. 3NF fixes transitive deps. BCNF fixes leftover non-superkey determinants.

📖 Concept Explained
  • 1NF: every attribute atomic (no lists/sets/repeats).
  • 2NF: 1NF + no partial dependency on candidate key.
  • 3NF: 2NF + no transitive dependency on key.
  • BCNF: 3NF + every non-trivial FD's LHS is a superkey.
  • Lossless join + dependency preservation required at each step.
🔗 Connects to FDs Anomalies Candidate keys Decomposition
WORKED EXAMPLE — Student/Course table
💡 Starting table — STUDENT_COURSE (UNF)

Attributes: StuID, StuName, Address, Courses (multi-valued), InstructorID, InstructorName, Dept.

UNF — STUDENT_COURSE (unnormalized):
+-------+-------+---------+----------------+--------+----------+------+
| StuID | Name  | Address | Courses        | InstID | InstName | Dept |
+-------+-------+---------+----------------+--------+----------+------+
| 101   | Anil  | Delhi   | DBMS, Networks | I1, I2 | Ravi,Sen | CSE  |
| 102   | Bina  | Mumbai  | DBMS           | I1     | Ravi     | CSE  |
+-------+-------+---------+----------------+--------+----------+------+

Given FDs:
  StuID            → StuName, Address
  StuID, Course    → InstID
  InstID           → InstName, Dept
Candidate key: {StuID, Course}
STEP 1 — Convert to 1NF (atomize)

Split multi-valued cells into separate rows. Every attribute now atomic.

1NF — STUDENT_COURSE:
+-------+------+---------+----------+--------+----------+------+
| StuID | Name | Address | Course   | InstID | InstName | Dept |
+-------+------+---------+----------+--------+----------+------+
| 101   | Anil | Delhi   | DBMS     | I1     | Ravi     | CSE  |
| 101   | Anil | Delhi   | Networks | I2     | Sen      | CSE  |
| 102   | Bina | Mumbai  | DBMS     | I1     | Ravi     | CSE  |
+-------+------+---------+----------+--------+----------+------+
Problem: Name, Address depend on StuID alone (partial dep on composite key).
STEP 2 — Convert to 2NF (remove partial deps)

Non-key attribute must depend on FULL key, not part of it. Split out partial deps.

2NF decomposition:

STUDENT(StuID, Name, Address)
+-------+------+---------+
| StuID | Name | Address |
+-------+------+---------+
| 101   | Anil | Delhi   |
| 102   | Bina | Mumbai  |
+-------+------+---------+

ENROLL(StuID, Course, InstID, InstName, Dept)
+-------+----------+--------+----------+------+
| StuID | Course   | InstID | InstName | Dept |
+-------+----------+--------+----------+------+
| 101   | DBMS     | I1     | Ravi     | CSE  |
| 101   | Networks | I2     | Sen      | CSE  |
| 102   | DBMS     | I1     | Ravi     | CSE  |
+-------+----------+--------+----------+------+
Problem in ENROLL: InstID → InstName, Dept (transitive dep through InstID).
STEP 3 — Convert to 3NF (remove transitive deps)

Non-prime attribute must not depend on another non-prime attribute. Split out InstID's deps.

3NF decomposition:

STUDENT(StuID, Name, Address)        — unchanged
ENROLL(StuID, Course, InstID)
+-------+----------+--------+
| StuID | Course   | InstID |
+-------+----------+--------+
| 101   | DBMS     | I1     |
| 101   | Networks | I2     |
| 102   | DBMS     | I1     |
+-------+----------+--------+

INSTRUCTOR(InstID, InstName, Dept)
+--------+----------+------+
| InstID | InstName | Dept |
+--------+----------+------+
| I1     | Ravi     | CSE  |
| I2     | Sen      | CSE  |
+--------+----------+------+
Check BCNF: in ENROLL suppose Course → InstID (one course = one instructor).
LHS Course is NOT a superkey of ENROLL  ⇒  violates BCNF.
STEP 4 — Convert to BCNF

Every non-trivial FD's LHS must be a superkey. Decompose ENROLL on Course → InstID.

BCNF decomposition:

STUDENT(StuID, Name, Address)
INSTRUCTOR(InstID, InstName, Dept)
COURSE_INST(Course, InstID)        ← from FD Course → InstID
+----------+--------+
| Course   | InstID |
+----------+--------+
| DBMS     | I1     |
| Networks | I2     |
+----------+--------+
ENROLL(StuID, Course)              ← residual link
+-------+----------+
| StuID | Course   |
+-------+----------+
| 101   | DBMS     |
| 101   | Networks |
| 102   | DBMS     |
+-------+----------+

✓ Every FD's LHS is now a superkey. Schema is in BCNF.
✓ Lossless join: natural-join all 4 tables reconstructs original 1NF table.
Pitfall

Always verify lossless join after each decomposition — common source of marks loss.

🎯 Mnemonic — "A-P-T-K": 1NF Atomic, 2NF no Partial, 3NF no Transitive, BCNF LHS=Key.
✅ Model Exam Answer
  1. Define each NF (1-line rule).
  2. Show starting table + given FDs + candidate key.
  3. 1NF: atomize multi-valued cells.
  4. 2NF: identify partial deps → split out.
  5. 3NF: identify transitive deps → split out.
  6. BCNF: check every FD's LHS is superkey → split if not.
  7. Verify lossless join + state final relations.

Q4 Closure of a Set of FDs — Compute F⁺ & Candidate Keys

⚡ Core Intuition

Closure X⁺ = all attributes you can reach starting from X using the given FDs.

If X⁺ = entire relation, then X is a superkey.

📖 Concept Explained
  • Attribute closure X⁺: set of all attributes functionally determined by X.
  • Use Armstrong's axioms iteratively until no new attribute added.
  • Candidate key = minimal superkey (no subset is a superkey).
  • Closure also used to check FD implication and equivalence of FD sets.
🔗 Connects to Armstrong Superkey BCNF check
WORKED EXAMPLE
R(A, B, C, D, E)
F = { A → BC,  CD → E,  B → D,  E → A }

Find A⁺:
  Step 0:  A⁺ = {A}
  Step 1:  A → BC      ⇒  A⁺ = {A, B, C}
  Step 2:  B → D       ⇒  A⁺ = {A, B, C, D}
  Step 3:  CD → E      ⇒  A⁺ = {A, B, C, D, E}
  Step 4:  no new      ⇒  STOP

  A⁺ = {A, B, C, D, E}  →  A is a SUPERKEY.

Check minimality:  {A} cannot be reduced → A is a CANDIDATE KEY.

Also: E → A → BC → D ⇒ E⁺ = ABCDE → E is also a candidate key.
      CD⁺  = CDE…AB    → CD is a candidate key.
      B⁺   = BD         → B alone is NOT a key.
🎯 Mnemonic — start with X, keep applying matching LHS, stop when stable.
✅ Model Exam Answer
  1. Define closure X⁺.
  2. State algorithm (initialise → loop FDs → fixed point).
  3. Show steps with each FD application.
  4. Conclude superkey / candidate key status.

Q5 4NF (MVD) and 5NF (Join Dependency)

⚡ Core Intuition

4NF stops "independent multi-values" from forcing combinatorial rows.

5NF stops "3-way (or higher) join dependencies" that single-key FDs can't capture.

📖 Concept Explained
  • MVD X ↠ Y: for each X, set of Y values is independent of other attributes.
  • 4NF: in BCNF + no non-trivial MVD whose LHS isn't a superkey.
  • JD ⋈(R₁,R₂,…): R = lossless join of its projections.
  • 5NF (PJ-NF): in 4NF + every JD is implied by candidate keys.
🔗 Connects to BCNF MVD JD Lossless join
🎯 Mnemonic — 4NF kills "↠", 5NF kills "⋈"-only redundancy.

  sid     INT PRIMARY KEY,
  name    VARCHAR(40) NOT NULL,
  dept_id INT REFERENCES dept(did),

SQL DML — SELECT / INSERT / UPDATE / DELETE U3

-- SELECT
SELECT col1, col2
FROM   table
WHERE  condition
GROUP  BY col1
HAVING aggregate_cond
ORDER  BY col1 [ASC|DESC]
LIMIT  n;

-- INSERT
INSERT INTO student (sid, name, cgpa) VALUES (101, 'Asha', 8.5);

JOIN Syntax (memorize all 5) U3

-- INNER JOIN  → rows matching in BOTH
SELECT s.name, c.title
FROM   student s INNER JOIN course c
  ON s.sid = c.sid;

-- LEFT (OUTER) JOIN → all left + matched right (NULL else)
SELECT s.name, c.title
FROM   student s LEFT JOIN course c ON s.sid = c.sid;

-- RIGHT (OUTER) JOIN → all right + matched left
SELECT s.name, c.title
FROM   student s RIGHT JOIN course c ON s.sid = c.sid;

-- FULL OUTER JOIN → all rows from both sides
SELECT s.name, c.title
FROM   student s FULL OUTER JOIN course c ON s.sid = c.sid;

-- NATURAL JOIN → auto-match on same-named columns
SELECT * FROM student NATURAL JOIN dept;

-- CROSS JOIN → Cartesian product (rare; avoid)
SELECT * FROM student CROSS JOIN dept;

Aggregate Functions + GROUP BY / HAVING U3

-- Aggregates: COUNT, SUM, AVG, MIN, MAX
SELECT dept_id,
       COUNT(*)   AS n_students,
       AVG(cgpa)  AS avg_cgpa,
       MAX(cgpa)  AS top_cgpa
FROM   student
GROUP  BY dept_id
HAVING COUNT(*) > 5;          -- HAVING filters groups
                              -- WHERE filters rows BEFORE grouping

-- DISTINCT
SELECT DISTINCT dept_id FROM student;

-- Subquery
SELECT name FROM student
WHERE cgpa > (SELECT AVG(cgpa) FROM student);

Views U3

-- CREATE
CREATE VIEW high_cgpa AS
  SELECT sid, name, cgpa FROM student WHERE cgpa >= 8.0;

CREATE OR REPLACE VIEW v_dept AS
  SELECT d.dname, COUNT(s.sid) FROM dept d
  LEFT JOIN student s ON d.did = s.dept_id GROUP BY d.dname;

-- USE
SELECT * FROM high_cgpa;

-- DROP
DROP VIEW high_cgpa;

-- Why: security, abstraction, simplify queries, logical independence.

Functional Dependency Notation + Closure U4

-- Notation
X → Y     reads "X determines Y"
            (whenever 2 tuples agree on X, they agree on Y)

-- Armstrong's axioms (sound + complete)
  Reflexivity:     if Y ⊆ X, then X → Y
  Augmentation:    if X → Y, then XZ → YZ
  Transitivity:    if X → Y and Y → Z, then X → Z
  (derived) Union:        X → Y, X → Z  ⇒  X → YZ
  (derived) Decomposition: X → YZ        ⇒  X → Y, X → Z
  (derived) Pseudo-trans:  X → Y, WY → Z ⇒ WX → Z

-- Closure { X }+ algorithm
1. result = X
2. repeat:
     for each FD  α → β  in F:
        if α ⊆ result :  result = result ∪ β
   until result stops growing
3. return result    -- if {X}+ = all attrs, X is a super-key

-- Normal forms (one-liner)
1NF : atomic attributes only
2NF : 1NF + no partial dep on candidate key
3NF : 2NF + no transitive dep (non-key → non-key)
BCNF: for every FD X→Y, X is a super-key

ACID + 2PL U4

ACID
  Atomicity   — all-or-nothing
  Consistency — DB stays in valid state
  Isolation   — concurrent T's don't see each other's partial work
  Durability  — committed changes survive crash

2PL protocol
  Phase 1 (Growing)   : acquire locks, release NONE
  Phase 2 (Shrinking) : release locks, acquire NONE
  ⇒ guarantees conflict serializability

Strict 2PL : hold ALL exclusive locks until commit/abort
           ⇒ avoids cascading rollback, ensures recoverability

What is degree of a relation?
show

Number of attributes (columns) in the relation.

What is cardinality of a relation?
show

Number of tuples (rows) currently in the relation.

Define Primary Key.
show

Minimal set of attributes that uniquely identifies each tuple; cannot be NULL.

Define Foreign Key.
show

Attribute in one relation referencing the primary key of another (or same) relation.

Difference between Candidate and Super Key?
show

Super key = any unique set; candidate key = minimal super key.

What is Data Independence?
show

Ability to change schema at one level without changing schema at the next higher level.

Logical vs Physical Data Independence?
show

Logical = change conceptual schema, external untouched. Physical = change internal, conceptual untouched.

What is a View in SQL?
show

Virtual table defined by a stored SELECT query; used for security and abstraction.

Define Functional Dependency.
show

X → Y: whenever two tuples agree on X, they agree on Y.

State 1NF.
show

Every attribute holds atomic (indivisible) values — no repeating groups.

State BCNF.
show

For every non-trivial FD X→Y in R, X must be a super key of R.

What is Concurrency Control?
show

Set of techniques (locking, timestamps) that ensure isolated execution of concurrent transactions.

State the 4 ACID properties.
show

Atomicity, Consistency, Isolation, Durability.

What is a deadlock?
show

Two+ transactions waiting indefinitely for locks held by each other.

Wait-Die vs Wound-Wait?
show

Both use timestamps. Wait-Die: older waits, younger dies. Wound-Wait: older wounds (preempts) younger.

Difference between DBMS and File System?
show

DBMS gives concurrency, integrity, security, query language, recovery; files give none.

What is an Entity Set?
show

Collection of similar entities sharing the same attributes (e.g. all Students).

Difference between Relational Algebra and Calculus?
show

Algebra = procedural (how). Calculus = declarative (what).

What does a checkpoint do?
show

Writes log+dirty buffers to disk and marks a recovery point so log scan starts later.

OLTP vs OLAP?
show

OLTP = many short transactions, normalized, current data. OLAP = few long analytical queries on warehouses.

Common Mistakes & Prof Red Flags

Read this 5 minutes before entering the hall. Each costs 1–3 marks.

  • Mixing primary key and candidate key: Candidate keys are all minimal super keys; primary key is the chosen one. Don't write them as synonyms.
  • ER diagrams without cardinality & participation: Always label 1:1 / 1:N / M:N and use double line for total participation. Missing these = no full marks.
  • BCNF "decompose" answers without lossless-join proof: Examiner expects you to show the common attribute is a super-key of one side.
  • Writing FD X→Y "means Y depends on X": Wrong direction in English. Correct: "X functionally determines Y" — Y is the dependent.
  • Confusing WHERE and HAVING: WHERE filters rows before grouping; HAVING filters groups after. Aggregates allowed only in HAVING.
  • NATURAL JOIN ≠ INNER JOIN: Natural auto-matches same-named columns and removes duplicate; inner needs explicit ON.
  • Calling 2NF/3NF "first/second/third level": They are forms, not levels. Each is defined by what it eliminates (partial / transitive dep).
  • Drawing precedence graph without labelling edges: Label each edge Ti → Tj with the conflicting operation. Cycle present = NOT conflict serializable.
  • 2PL diagram missing "no release in growing": The growing phase must show only acquires, shrinking only releases. Mixing = wrong protocol.
  • Forgetting NULL handling: NULL ≠ 0, NULL ≠ ''. Comparisons with NULL yield UNKNOWN — use IS NULL / IS NOT NULL.
  • Writing SELECT * for every query: Costs precision marks. Always project only requested columns.
  • Skipping the definition line: Every long answer must start with a 1-line crisp definition before diagram. No definition = lose 1 mark even if rest is perfect.
